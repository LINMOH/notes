        - 基本策略
          - 最优匹配
            - 核心思想
              - 从所有足够大的块中，选出最小的那块
            - 执行过程
              - 必须遍历整个空闲列表
            - 优点
              - 尽量保留更大的空闲块，不浪费大空间
            - 缺点
              - 性能差
              - 外部碎片
          - 最差匹配
            - 核心思想
              - 从所有块中，选出最大的那块
            - 执行过程
              - 必须遍历整个空闲列表
            - 优点
              - 分割后剩下的块依然很大，大概率能满足后续的其他请求
            - 缺点
              - 性能差
              - 过度消耗
          - 首次匹配
            - 核心思想
              - 找到第一个满足大小要求的块就停下
            - 执行过程
              - 从头开始扫描，命中即返回
            - 优点
              - 速度快，不需要遍历全表
            - 缺点
              - 空闲列表的前端会被频繁分割，产生大量碎片
              - 通常配合“按地址排序”使用，以便于后续的合并
          - 下次匹配
            - 核心思想
              - 在首次匹配的基础上，记住上次查找的位置
            - 执行过程
              - 下一次申请从上次结束的地方继续向后查找
            - 优点
              - 避免了碎片在列表头部堆积，将分割压力分散到整个堆中
              - 速度快，不需要遍历全表、
            - 性能与首次匹配接近，但在某些工作负载下碎片分布更均匀
          - 分离空闲列表
            - 核心逻辑
              - 为常用大小的对象（如锁、inode）建立专属的空闲列表
            - Slab 分配器
              - 对象缓存
                - 内核启动时为频繁请求的对象创建特定大小的缓存
              - 厚块
                - 当缓存用尽，向通用分配程序申请一大块内存（页的公倍数）
              - 状态保持
                - 空闲对象保持在预初始化状态，避免重复调用构造/析构函数的巨大开销。
            - 优点
              - 零碎片
                - 同类对象排列紧密，消除外部碎片
              - 极速
                - 无需复杂的列表搜索，直接从专属池中取用
            - 回收机制
              - 当 Slab 内对象引用计数归零且系统急需内存时，通用分配程序可回收该空间
          - 伙伴系统
            - 核心逻辑
              - 将内存空间看作 $2^N$ 的大块，通过递归对半拆分来满足请求
            - 分配流程
              - `64KB`
                - `32KB`
                  - `16KB`
                    - `8KB`
                    - `8KB`
                - `32KB`
                  - `16KB`
                    - `8KB`
                    - `8KB`
            - 合并机制
              - 伙伴定义
                - 物理地址相邻且大小相同的两块（其地址仅在二进制的一位上不同）
              - 递归合并
                - 当 `8KB` 块归还时，检查其“伙伴”是否也空闲。若是，则合并为 `16KB`，再检查 `16KB` 的伙伴，以此类推直到无法合并
            - 优点
              - 合并操作极其简单、高效
            - 缺点
              - 内部碎片严重（若申请 33KB，必须分配 64KB）
